# VisionPaint JWT Auth

**Status:** Replaces cookie + antiforgery (2026-05-17).

## Why

- Firebase Hosting and Azure API are cross-origin; third-party cookies fail in Incognito.
- Mobile apps need `Authorization: Bearer` without browser cookie rules.

## Flow

1. `POST /api/auth/login` or `/bootstrap` -> `accessToken` (15 min) + `refreshToken` (14 days).
2. Client sends `Authorization: Bearer {accessToken}` on API calls.
3. On 401, `POST /api/auth/refresh` with `refreshToken` -> new pair and revokes the previous refresh token.
4. Reusing a rotated refresh token revokes the rest of that refresh-token session chain.
5. `POST /api/auth/logout` revokes the active refresh-token session and the client clears stored tokens.

## Configuration

| Key | Where |
|-----|--------|
| `VISIONPAINT_JWT_SIGNING_KEY` | Azure App Settings (required, >=32 chars) |
| `Jwt:*` | `appsettings.Production.json` (issuer, audience, lifetimes) |

## Frontend

Tokens in `sessionStorage` under `visionpaint.auth`. No `credentials: include`, no CSRF header.

## Mobile (future)

Same endpoints: store refresh token in secure storage (Keychain/Keystore), access token in memory.
